# 人生第一份 Issue 生成记：从一次 AI 工具链故障到开源反馈

> **日期**: 2026-05-27  
> **主题**: AI 工具链故障、网络排障、开源协作、个人成长  
> **状态**: 完成  

> 这一篇不只是“技术故障复盘”，更像是作为一个长期自学者，第一次真正参与开源协作的节点记录。这是一篇偏叙事、复盘与成长感并存的深度技术日志。

## 背景

今天发生了一件对我很有意义的事：我第一次向一个开源项目正式提交了 Issue。

今天一到办公室，想起昨天的 VPS 整理，想继续这个工作。打开 NetCatty，提示有更新，当然启动更新，以为又会有新的能力提升，更新之后满怀期待。但事情并不浪漫，甚至一开始有点狼狈。我打开 NetCatty，准备使用 AI 辅助功能，却突然遇到报错：

`Failed after 3 attempts. Last error: connect ETIMEDOUT 10.0.0.1:443`

一开始我完全没反应过来这意味着什么。以为是 AI 上游 provider 的问题，切换模型，没有反应，难道是 AI 模型今天集体罢工？想着“有问题、找 Codex”，期待 Codex 能够解决。

更麻烦的是，不只是 NetCatty 出问题，连我平时最信赖的 Codex 也联系不上了。Claude Desktop 也长时间没有反应。

这让我一度措手无策。

平时遇到问题，我已经习惯让 Codex 帮我冷静分析、拆解证据、判断风险。但这一次，连“参谋长”都失联了。那种感觉很特别：不是单个应用坏了，而是整个 AI 工具链突然塌了一块。

## 初步求助：从 DeepSeek 网页版获得基本判断

幸好浏览器还正常，VS Code 也没有受到影响。我先通过网页版 DeepSeek 求助。

DeepSeek 给出的第一层判断是：`ETIMEDOUT` 表示连接超时，而 `10.0.0.1:443` 是内网地址，不像是真正的 AI 服务地址。它提示我重点检查代理、VPN、hosts、DNS、网络隔离等方向。

这个判断虽然比较泛化，但很重要。它让我意识到：问题大概率不是 AI 服务本身坏了，也不是账号问题，而是请求被错误地导向了一个内网地址。

这一步建立了第一个关键判断：

> `10.0.0.1:443` 很可能是错误代理目标。

## 关键排障：VS Code Copilot 帮我把问题救回来

接下来，我通过 VS Code Copilot 做了更具体的排障。——万幸，这里还像平时一样可以正常工作。也给自己提个醒：在这个 AI 时代，真的得多几个通道，才能多几条生路！

Copilot 帮我检查了：

- `ipconfig`
- `route print -4`
- WinHTTP 代理
- git / npm 代理
- WinINET 注册表代理
- VS Code 用户设置
- `10.0.0.1:443` 和 `192.168.50.1:443` 的连通性

结果逐渐清晰：

- 我的真实网段是 `192.168.50.0/24`
- 默认网关是 `192.168.50.1`
- `10.0.0.1:443` 对当前机器不可达
- WinHTTP、npm、git、系统代理都没有明显污染
- 我当前的网络架构是华硕路由器 + MerlinClash 透明代理，本机不应该配置显式代理

这时问题就基本定位了：

> 某个应用把 AI 请求错误地指向了 `10.0.0.1:443`，而这个地址在我的网络环境中并不是可用代理。

Copilot 帮我修改 NetCatty 的代理 / 网络设置，让它回到直连、跟随系统或默认模式。重启后，NetCatty 恢复了。

哈哈哈，更关键的是，Codex 和 Claude Desktop 也陆续恢复响应。

## 回到 Codex：复盘、审计与确认影响面

恢复 Codex 后，我没有马上把事情翻篇，而是继续复盘。

Codex 帮我做了几件事：

1. 梳理本机网络基线：我的代理层在路由器，而不是 Windows 本机。
2. 检查系统代理、npm、git、VS Code、Cursor、Codex、Claude 的本地配置。
3. 查 NetCatty 后台目录和更新记录。
4. 确认 NetCatty 当天发生了 `1.1.14 -> 1.1.15` 更新。
5. 发现 NetCatty updater 在 08:38-08:39 准备了 `Netcatty-1.1.15-win-x64.exe`。
6. 发现 vault backup 记录中存在 `reason=app_version_change`。
7. 结合 Codex 日志，确认在异常窗口内 Codex 也出现过等待超时。

最终判断是：

> 这不是系统级代理污染，而是 NetCatty 更新后，高概率重新激活、迁移或恢复了某个错误的应用级代理 profile，导致 AI 辅助请求被指向 `10.0.0.1:443`。

这次故障影响不小。NetCatty 自己的 AI 辅助失效，Codex 一度联系不上，Claude Desktop 也长时间没有反应。幸好浏览器和 VS Code 没有受影响，我才有机会借助 DeepSeek 网页版和 VS Code Copilot 把问题救回来。

这也给我一个很重要的提醒：

> AI 工具链不能只有一个入口。  
> 当最信赖的工具失联时，必须还有备用通道。

## 提交 Issue：第一次正式向开源项目反馈

后来我查了 NetCatty 的更新说明，发现 v1.1.15 里确实包含不少 AI / Agent / Claude / Codex 相关改动，例如：

- `[codex] Fix Claude ACP Windows shim launch`
- `Per-agent provider switcher chip in the Catty Agent chat input`
- `Serialize Catty terminal_execute per session`
- `Suppress SDK reasoning/text state-machine error chunks`

这让我意识到，这次问题不是一个孤立的小毛病。它有可能涉及 NetCatty 更新后对 AI / Agent / provider / proxy 状态的处理方式。

于是，我决定向 NetCatty 原作者提交 Issue。

这是我人生第一次完整提交 Issue。

我没有写成情绪化抱怨，也没有暴露太多本机隐私细节，而是尽量客观描述：

- 更新后出现 `10.0.0.1:443` 超时
- 我的网络是路由器透明代理，本机无显式代理
- 修复 NetCatty 网络 / 代理设置后恢复
- 不只 NetCatty AI Assistant 受影响，Codex 和 Claude Desktop 也曾无响应
- 希望作者检查更新时是否过度恢复或激活了旧 proxy profile

提交之后，我还追加了临时修复方式：

> Changing the proxy/network setting back to direct/system/default mode and restarting NetCatty fixed the issue for me.

这件事让我有一种小小的成就感。

不是因为我“解决了一个 bug”，而是因为我终于从一个只会使用工具的人，迈出了一步，开始把自己的真实问题反馈给开源项目。

## 更长的时间线：从 2017 到今天

这件事对我有意义，还因为它不是突然发生的。

我不是科班开发者。更准确地说，我是一个凭兴趣进入这个领域的人。

2017 年冬天，我第一次注册并登录 GitHub。那时候很多东西都看不懂，只是觉得这里像一个巨大的技术森林。后来一点点摸索，fork 项目，读 README，改配置，试脚本，踩坑，重装，搜索，问 AI，写文档。

到今天，我已经 fork 或创建了 138 个 repositories。

这个数字不一定代表“我多会开发”，但它确实记录了我这些年的探索轨迹：

- 我学习过很多工具。
- 我尝试过很多方向。
- 我启动过很多项目。
- 我也放弃过很多东西。
- 我越来越能从混乱中识别结构。
- 我越来越能把一次故障变成一份经验。

今天第一次提交 Issue，让我突然意识到：

> 原来我已经不只是站在开源世界外面观看。  
> 我也可以把自己的真实使用经验，反馈回这个世界。

哪怕只是一个小 Issue，哪怕只是一次应用更新后的网络异常，它也意味着我和开源生态之间有了一次真实互动。

## 这次学到的东西

这次事件教会我的，不只是 NetCatty 的代理设置。

更重要的是几种能力：

1. 在恐慌中抓住关键报错。
2. 区分系统代理、应用代理和路由器透明代理。
3. 借助不同 AI 工具交叉验证，而不是依赖单一工具。
4. 把“感觉像是更新导致”转化为时间线证据。
5. 区分直接原因、高概率诱因 and 未证明假设。
6. 给开源项目提交克制、清晰、有价值的 Issue。
7. 保持备用通道，不让单点故障瘫痪整个 AI 工作流。

尤其重要的是：这次我真正体验到了“排障闭环”。

不是简单地修好就算了，而是：

发现问题 → 外部求助 → 本地验证 → 恢复服务 → 复盘原因 → 确认影响面 → 写入记录 → 提交 Issue → 追加 workaround。

这是一条完整链路。

## 后续提醒

以后 NetCatty 更新要进入高风险观察名单。

特别是涉及 AI Agent、provider、Claude、Codex、ACP、proxy profile、网络设置的更新，不能再默认信任。更新前最好保留配置快照，更新后做一次快速巡检。

同时，我也要保留浏览器、VS Code、Codex、Claude、DeepSeek 等多条工具通道。今天能解决问题，恰恰是因为浏览器和 VS Code 没有一起失效。

这次经历也提醒我：

> AI 时代的工具链越强，越需要治理能力。  
> 工具会失效，入口会中断，配置会漂移。  
> 真正可靠的不是某一个工具，而是自己逐渐建立起来的判断、记录和复盘能力。

## 结语

今天这份 Issue，也许作者会看到，也许不会。也许它会被修复，也许只是留在 GitHub 的某个角落。

但对我来说，它已经有意义了。

这是我人生第一份 Issue。

它来自一次真实故障，一次措手无策，一次多工具协作排障，一次重新恢复秩序的过程。

从 2017 年冬天第一次登录 GitHub，到今天第一次完整提交 Issue，我好像终于在这个开源世界里留下了一个小小的脚印。

不是很大，但是真实。

这就够了。
