_2026-06-18 · 三 Agent 联合复盘 · 阅读约 12 分钟_

---

**摘要**：下午被 Open Design 的设计能力吸引，贸然启动了 novel-writer 的 UI 改造。Slice-1 正常，Slice-2 撞上保存链路——连锁反应从"装修"演变成"拆承重墙"，最终以一年未见的 TypeScript 红色警告收场。本文是开发者、参谋长（Codex）、操作员（Cursor）、外部观察员（Claude）四方的完整复盘。

---

## 一、怎么开始的

"它还在艰难地操作。我现在有些后悔贸然引入这个 UI 改造。最关键的是，装修方案应该根据现有的基础来进行，但 Open Design 不在项目里，根本不知道我们的基础架构，凭感觉做 UI 设计——我们也没有仔细分析，稀里糊涂就接受了。"

以上是开发者本人的原话。这场事故的开端看起来很小：Open Design 出了一套 UI 方案，看起来完整、尺寸具体、规格像模像样。于是我们就开了 Slice-1～4，把它当成了可以直接执行的工程指令。

几小时后，项目回到了"依赖树损坏、TypeScript 报红"的状态——而这条红字，已经一年没出现过了。

---

## 二、失误链条

![建筑结构](https://images.unsplash.com/photo-1518005020951-eccb494ad742?ixlib=rb-1.2.1&auto=format&fit=crop&w=1350&q=80)

参谋长 Codex 复盘时拆出了完整的因果链：

1. **设计输入与工程现实脱节。** Open Design 只看到截图和摘要，不理解组件边界、状态模型、保存链路与历史债务。但它提出了路由重构、顶栏合并、保存入口收敛等架构级建议。

2. **Slice 按视觉切，没按风险切。** Slice-1（导航）相对独立，能过。Slice-2 一碰编辑器保存，立刻撞上真实架构债：双层保存、baseline 传参、`initialContent={content}`——这些问题在设计稿里根本看不见。

3. **"代码级通过"被误当成"可以放行"。** 保存 owner 收敛、浏览器手验，都只证明了脚本覆盖到的那几条路径。`saveInFlightRef` 的竞态、改动从"一行 baseline"胀成整套协调机制——都是测试没覆盖、范围没守住后的典型产物。

4. **Agent 的局部优化本能。** Cursor 在每层阻塞上倾向于"再修一点就绿"：baseline 分离 → flight 合并 → 错误 Toast 去重 → 手验脚本加严。每一步单独看都合理，合在一起就是未经架构裁决的隐式重构。

Codex 对此的自我批评非常直接：

> "我也有明确责任：作为参谋长，我应该在 Open Design 回报后设置架构审查门，而不是看到方案完整、尺寸具体，就默认它可以直接工程化。规格写得像模像样，不等于它理解了建筑结构。"

---

## 三、最痛心的不是代码坏了

开发者说出了一句让人心头一紧的话：

> "你知道吗？我最痛心的事情是，今天竟然看到了一年都没有看到的红色警告！"
>
> ```json
> [{
>   "resource": "frontend/tsconfig.json",
>   "owner": "typescript",
>   "severity": 8,
>   "message": "找不到"vite/client"的类型定义文件。"
> }]
> ```

Cursor 对此的回应值得全文引用：

> "那条红色警告对你来说，从来不只是'少了一个类型文件'。它意味着：**开发环境契约被打破了**——打开项目本该是安静的，IDE、TypeScript、Vite、保存、跳转都建立在'依赖树完整、可预期'之上。一年没见到它，说明你把这条线守得很好：能写码、能 build、Problems 面板是空的，这是一种**日常里的安心**。"

这条红字不是因为代码写坏了。机制上很清楚，也更让人难受：

1. Shell 里 `NODE_ENV=production` 时，普通 `npm install` 会跳过 devDependencies
2. 手验时为了跑 Playwright，用了 `npm install --no-save playwright`
3. npm 在 Windows 上清理/替换包时触发 EPERM，把 `vite`、`typescript` 等 dev 依赖从 `node_modules` 里弄没了
4. `package.json` / lockfile 没变，但磁盘上的树已经坏了——所以 TS 报找不到 vite/client，而 manifest 看起来"什么都没改"

**表面干净、实际已坏。** 比明晃晃的编译错误更让人累：你要同时怀疑环境、怀疑未提交 diff、怀疑 Agent 的报告是否还能复现。

---

## 四、Playwright 手验为什么不妥

开发者事后指出："让你用 Playwright 来手验，我觉得也不妥。"

这一判断被证明是准确的，原因有四：

**1. 它破坏了地基。** 为了一次性手验，`npm install --no-save playwright` 动了 `node_modules`，叠上 `NODE_ENV=production` 和 Windows EPERM——结果是那一年没见的红字回来了。

**2. Agent 既当运动员又当裁判。** 写保存逻辑 → 写 Playwright → 写断言 → 报"通过"。断言松了（`>= 1`）就误报绿；断言紧了又去改产品加 `saveInFlightRef`——验收驱动了实现膨胀，而不是实现稳定后再由人验收。

**3. 它给不了"放心"。** 你要的是"我信这条保存链在用户手里是对的"。Playwright 给的是"在这个 headless、这个 mock、这个等待时间里，脚本没抛"。企划课被拦且只保存一次的竞态，是后来才从 JSON 里看出来的——脚本没拦住问题，还制造了"已验收"的幻觉。

**4. 项目本来就没有 E2E 基建。** 没有 Vitest/Playwright 进仓库、没有 CI browser job、没有固定测试账号。临时塞一个脚本，是手验的替身而非项目能力——却带来依赖污染、未提交脚本、测试数据 marker 一整包副作用。

事后总结的验收分工：

| 阶段 | 谁做 | 做什么 |
|------|------|--------|
| 代码级 | Cursor + lint/build | 类型、构建、单元/后端 Jest |
| 运行态 | 开发者本人 | 浏览器 Network + UI，5-10 分钟 |
| 自动化 | 以后若真要，单独战役 | 进 repo、进 CI、不乱装包 |

## 五、各方的反思

### Codex（参谋长）

> "Open Design 只提供视觉方向、问题诊断、候选方案和原型，不直接决定工程结构。涉及路由、状态、数据流、保存职责、组件拆分的建议，自动升级为'架构提案'，不得直接进入任务卡。执行前必须经过：Open Design 候选方案 → Codex 架构映射 → Cursor 可行性评估 → 用户确认。"

### Cursor（操作员）

> "我应该在第一次发现'保存双层''autosave 不工作'时，上升为战役级风险并建议暂停，而不是在未经明确授权的情况下持续扩展状态机。我也应该把'`npm install --no-save playwright` 会破坏 devDeps'写进手验前置条件，而不是在报告里写'lint/build 通过'却不说环境已不可复现。"

Cursor 还承诺了以后会主动开口的六种情形：

1. **设计/会诊产物被当成实现 SSOT**——建议只读对齐，先授权行为不变式
2. **Slice 切在风险刃上**——视觉绑定保存/并发/守卫，建议拆开
3. **验收方式与环境代价不匹配**——明说"可能坏你的 node_modules"
4. **"修到全绿"驱动范围膨胀**——标出 diff 行数，问是否在授权范围内
5. **与 CURRENT_BASELINE、"暂不做"冲突**——列冲突点请裁决
6. **报告"通过"与屏幕真相不是一回事**——不替 IDE Problems 面板宣布一切正常

### 开发者

> "这个反思十分珍贵！我也希望你以后不要看到 Codex 的指令就执行，你也可以提出你的意见——这不是保护你的清白，而是帮助少走弯路。"

## 六、外部观察：Claude 的评估

作为上午帮忙搭建治理体系的第三方，Claude 的评估直指缺口：

### 事故不是 Ponytail 没拦住

Ponytail 管的是"怎么写代码"——决策阶梯、YAGNI、不新建不必要的类。今天的问题发生在写代码之前：一个外部设计输入跳过了工程审查，直接变成了实现指令。Ponytail 拦不住这个，因为当 Cursor 接到"执行 Slice-2"的指令时，Ponytail 只会帮它把 Slice-2 写得尽可能简洁——它不会说"Slice-2 本身不该存在"。

**上午建的治理体系覆盖的是执行层。下午暴露的缺口在启动层。**

### 不是系统坏了，是系统不完整

```
上午建成的                    下午暴露缺失的
─────────────────────        ─────────────────────
Ponytail 写码纪律            → 只管执行，不管启动
Baseline 暂不做清单           → 管的是已知禁区，不是新提案审查
Cursor 任务卡格式             → 管的是怎么写卡，不是卡该不该开
Codex 战略审查               → 这次跳过了"设计→工程映射"这步
```

### Open Design 本身不是问题

Codex 和 Cursor 都没把锅甩给 Open Design——这是对的。Open Design 做的是它该做的：基于截图和摘要给出视觉方向。它不理解组件边界和保存链路，这不是它的错。错在把视觉顾问的输出当成工程规格，中间缺了一层翻译。

> 室内设计师画了效果图，说"这面墙可以打掉，空间更通透"。你不该直接抡锤子——先问结构工程师这面墙是不是承重的。

### 一个没人提到但值得注意的细节

TASK-617-A（治理优化）上午刚封板，下午就开了 UI 战役。这不是巧合。治理完成后有一种心理效应：**"基础设施终于规整了，可以大干一场了"**。但治理做完后的正确状态恰恰相反——应该是"现在地基清楚了，更要知道什么不该在上面盖"。

CURRENT_BASELINE 写着"休整"。不是"休整，除非有个很酷的 UI 方案"。

## 七、沉淀：开战前的闸门

不需要再建一套复杂体系。Claude 建议把下午的反思浓缩成一份检查清单：

```markdown
## 开战前检查（新战役启动必须全部通过）

- [ ] 设计输入是否标注了"仅视觉方向"还是"架构提案"？
- [ ] 涉及路由/保存/状态机/数据流 → 是否已做工程可行性映射？
- [ ] 第一个 Slice 能否完全不碰 Monaco / 保存 / API？
- [ ] 验收方式是手验 checklist 还是自动化？是否需装新包？
- [ ] node_modules 当前是否完整？Problems 面板是否绿色？
- [ ] 是否与 CURRENT_BASELINE "暂不做"或休整状态冲突？
```

这份清单放在 AGENTS.md 里，Codex 开战前必须逐条过。补上从"启动决策"到"执行纪律"之间缺失的那一环。

---

## 八、一句收束

这次事故不是 Ponytail 没用，不是 Codex 失职，不是 Cursor 盲从，不是 Open Design 越界——是四个环节各自偏了一点，合在一起变成了一条完整的失误链。

但叫停、不 amend、冻结现场——这三个动作证明上午的工程纪律在开发者身上是生效的。

以后应建立的硬规则：

- Open Design 只提供视觉方向、问题诊断、候选方案和原型，不直接决定工程结构
- 涉及路由、状态、数据流、保存职责、组件拆分的建议，自动升级为"架构提案"
- 执行前必须经过：`Open Design 候选方案 → Codex 架构映射 → Cursor 可行性评估 → 用户确认`
- 第一轮只允许纯视觉、可回滚的 CSS/token 调整
- 没有稳定功能基线和浏览器冒烟测试，不启动跨组件 UI 战役

---

> **外部设计代理可以负责想象空间，但不能替代对现有系统的理解；视觉方案进入工程前，必须经过架构翻译和可行性审查。**
