# LifeVideo·Neo 开发手记：从混乱到有序的架构演进

**日期**: 2025-12-08
**记录人**: LifeVideo·Neo 架构团队

## 1. 缘起：不仅仅是写代码
今天的开发历程是一个教科书式的“从 Idea 到 MVP”的缩影。我们不仅仅是修复了几个 Bug，更是在实战中深刻体会到了**系统工程**的重要性。

## 2. 关键转折点 (The Pivots)

### 2.1 架构之争：拥抱标准 vs 依赖集成
*   **困境**: 起初我们使用了 `ModelScope` SDK，试图用“全家桶”解决问题。但很快陷入了 Python 3.13 的依赖地狱，`datasets` 库的冲突让我们寸步难行。
*   **顿悟**: “如无必要，勿增实体”。
*   **决策**: **果断切割**。我们转向了更底层的 `Transformers` + `PyTorch` 纯净方案。虽然代码需要多写几行（比如手动下载模型、手动写推理逻辑），但换来的是**绝对的掌控力**和**工业级的稳定性**。
*   **经验**: **技术选型不能只图上手快，更要看生态兼容性和底层可控性。**

### 2.2 本地化生存：网络环境的适配
*   **问题**: AI 模型下载在中国网络环境下极其缓慢。
*   **决策**: 引入 `HF-Mirror` 并将其写入代码基因 (`os.environ["HF_ENDPOINT"]`)。
*   **经验**: **优秀的软件必须尊重用户的运行环境，而不是强迫用户去适应软件。**

### 2.3 需求进化：从“功能”到“体验”
*   **MVP 1.0**: 能搜就行。结果 -> 搜“女孩”出来很多不相关的低分视频。
*   **MVP 1.5 (Pro)**: 发现“单帧盲区”和“低阈值”问题。提出了 **“三点采样法”**。
*   **MVP 2.0 (Smart)**: 发现 CLIP 很贵，且元数据被浪费了。提出了 **“元数据优先 (Metadata-First)”** 策略。
*   **经验**: **真正的需求往往不是最初想到的那个。** 用户真正要的不是“AI 搜索”，而是“高效找到记忆”。如果元数据能做到，就应该用元数据，而不是盲目迷信 AI。

### 2.4 范围控制：捡西瓜扔芝麻
*   **危机**: 随着讨论深入，我们想做溯源、查重、自动剪辑……项目面临严重的范围蔓延 (Scope Creep)。
*   **对策**: 引入 **P0/P1/P2 分级制度**。
    *   P0 (基石): 必须先保证搜得准 (多帧采样)。
    *   P1 (智能): 来源溯源。
    *   P2 (花样): 自动更名。
*   **经验**: **克制是架构师最美德。** 没有坚实的地基（搜索精度），上层的花哨功能由于沙上建塔，毫无意义。

## 3. 结语
今天的成果远不止是跑通的代码，更是明确了一套**“稳扎稳打”的方法论**：
1.  **架构先行**（纯净栈 > 臃肿栈）。
2.  **数据为王**（元数据 > 盲目 AI）。
3.  **节奏制胜**（P0 > P1 > P2）。

这段经历将被封存进项目的基因里，指引 LifeVideo·Neo 未来的每一次迭代。
