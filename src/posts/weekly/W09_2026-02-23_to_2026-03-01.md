### 1.1 核心目标 (Priority P0)

#### 1. BT-Player V1.1：全协议流媒体播放器 (✅ 里程碑达成)
**状态**: 🟢 已完成 (2026-02-24)
**结果**:
- **能力升级 (Magnet + HLS)**：从单一"磁力链播放器"进化为支持 Magnet + HLS (m3u8) 的综合隐私流媒体终端
- **关键技术攻关**：
    - `librqbit` API 升级适配：移除废弃配置项，完美兼容 9.0.0-beta.2
    - `mpv.lib` 链接修复：解决 Windows 下 `mpv.lib` 缺失/格式不兼容问题，通过 Python 动态生成 `.def` 导出表并构建 MSVC 导入库，彻底打通原生渲染链路
    - 构建链路优化：修正路径处理逻辑，支持一键 `cargo build` / `npm run tauri dev`
- **V1.1 核心能力**：
    - 高性能 BT 引擎：基于 Rust `librqbit`，DHT 快速解析 + 边下边播 + UPnP/NAT-PMP 自动内网穿透
    - 4K 级 GPU 渲染：通过 `libmpv` FFI 集成，`hwdec=auto-safe` 硬件加速
    - HLS 边播边录：m3u8 多码率自适应切换 + 一键录像 → 持久化为本地 `.ts`
    - 智能策略引擎：AI 启发式算法扫描磁力链文件树，自动标记广告/色情低质文件
    - 无痕本地存储：SQLite 记录播放历史与断点，零服务器依赖
- **当前状态**: `稳定 (Excellent)` — 22+ 人月等效工时压缩至 AI 协作高效完成

#### 2. cf-pikpak：三个版本连续发布 (✅ 密集迭代)
**状态**: 🟢 已完成 (2026-02-26 ~ 2026-02-27 连发三版)
**结果**:

- **v3.0.0 (2026-02-26)** 🎉 — *播放列表驱动 + HLS 全面支持*
    - `saveAndGetLink` 逻辑修复：支持多种 PikPak API 返回格式 (file.id / files[0].id / task_id)
    - 新增 `/api/proxy` 路由：通用流代理，支持 m3u8 URL 重写
    - M3U8 处理：重写 `#EXT-X-KEY`、`#EXT-X-MAP`、`.ts` 分片 URL，解决 CORS
    - 播放器双格式支持：自动识别 MP4 / HLS，动态 source type

- **v3.0.1 (2026-02-27)** — *本云盘分享 + HLS 检测修正*
    - 有 `file.id` 时直接跳转 `/play/{fileId}`，跳过 match-play 冗余调用
    - `files.js` 新增 `_probeIsHls(url)`：对 `download/?fid=...` 类 URL 做 HEAD 探测
    - 探测为 HLS 时自动走 `/api/proxy` 加载，解决误判为 MP4 的播放中断
    - 验证：516 MB 文件完整连续播放 ~24 分钟，零中断

- **v3.0.2 (2026-02-27)** — *生产部署 + 移动端优化*
    - **正式上线**：`https://pikpak-player.pennfly2008.workers.dev` 部署成功
    - **自定义域名**：`pikpak.f2008.tk` 可用，双入口验证通过
    - Token 修复：旧 Refresh Token 失效后通过 `get_token.py` 更新 Secret
    - 移动端首帧策略：手机/iPad 时优先级 HD > SD > FHD > ORIGIN（更省流量/更快首帧）
    - proxy 层增强：`isM3u8Content` 内容嗅探防误判 + `fetchWithRetry`（3次重试 + 指数退避：200ms/400ms/800ms + 30s 超时）

#### 3. AsusProxy：家庭网络重构 PRD (✅ 方案确立)
**状态**: 🟢 已完成 (方案文档落地)
**结果**:
- **MerlinClash 代理优化**：
    - 识别核心痛点：透明代理劫持内网流量 → NAS 服务 (SMB/FTP) 访问异常
    - 制定直连规则清单：`192.168.0.0/16`、`10.0.0.0/8` IP 段直连 + 关键服务端口直连 (445/SMB、21/FTP、8080/FileBrowser)
    - Cloudflare Tunnel 回环问题修复：`PROCESS-NAME,cloudflared,DIRECT` 规则
- **WireGuard VPN 底层方案**：
    - 策略：**"Clash 主内，WireGuard 主外"** — Clash 接管全屋智能分流，WireGuard 给 Apple TV / 游戏机等设备提供纯净 IP 通道
    - 配置 Surfshark SG (新加坡) + Surfshark US (洛杉矶) 双线路
    - VPN Director 设备级分流：指定设备流量强制走 WireGuard，与 Clash 完全隔离
- **BT-Player 专属网络规则**：
    - BT 监听端口固定 `50051` (TCP/UDP) + uTP 穿透
    - Clash 规则 `DST-PORT,50051,DIRECT` + `PROCESS-NAME,bt-player.exe,DIRECT`，确保 BT 流量不走代理、DHT 直连速度最大化
- **内存优化**：为路由器（RT-AX86U Pro / 梅林固件）创建 2GB USB Swap，解决多服务并发 OOM 问题

### 3.3 深度思考 & 洞察

#### [战略] 三个项目的生态协同效应
> **触发**: BT-Player 需要 Clash 规则，cf-pikpak 需要 HLS，AsusProxy 承载网络底座。

**核心洞察**:
- 超个体的真正威力不在单个项目的深度，而在于**多个项目之间的横向协同**。
- 本周的三条战线看似独立，实则相互咬合：BT-Player 的流量分发依赖 AsusProxy 的 Clash 规则；cf-pikpak 的 HLS 代理与 BT-Player 的 HLS 引擎共享同一套媒体生态逻辑。
- 这是从"点状能力"到"系统能力"的跃升。

#### [工程] Rust FFI 的"顿悟时刻"
> **触发**: 用 Python 脚本辅助解决 `mpv.lib` MSVC 导入库问题。

**核心洞察**:
- 当遇到底层链接库问题时，"绕过"往往比"正面硬刚"更优雅。动态生成 `.def` 导出表再用 `lib.exe` 构建导入库，是一种"外科手术式"的跨语言协作。
- AI 时代的非科班开发者优势：不被"这应该怎么做"的路径依赖束缚，而是聚焦"这个问题怎么解决"。

#### [反思] 节后复工节奏的校准

回顾来看，W09 的输出量超出预期——这证明 W08 的"磁盘碎片整理"是有效的。充分的休息确实会转化为更高质量的工程决策（如 cf-pikpak 的 HLS 内容嗅探方案设计清晰、一次到位）。
