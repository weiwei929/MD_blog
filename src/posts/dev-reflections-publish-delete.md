# 博客开发实录：从功能实现到体验打磨的思考

今天是非常充实的一天。我们集中攻克了博客系统的两个核心功能：**文章发布**与**文章删除**。

这看似简单的"增删"操作，背后却隐藏着不少交互细节的考量。在解决这些问题的过程中，我深刻体会到了开发不仅仅是写代码，更是一种对用户体验的深度共情和对架构的全局把控。

以下是我在今天的开发过程中的三点核心感悟。

## 一、 精益求精：细节决定产品的温度

在实现"删除文章"功能时，我们最初只是简单地调用了浏览器的 API。功能是跑通了，但体验却很生硬：浏览器弹出一个冷冰冰的框，问你要不要删除，紧接着又弹出一个框问你要权限。

**"能不能做得更好一点？"**

抱着这个想法，我们对交互进行了深度的打磨：

1.  **拒绝"盲删"**：我们在确认弹窗中，不仅显示了文章标题，还特意计算并显示了**具体的文件名**（如 `src/posts/my-article.md`）。这让用户清楚地知道，自己按下按钮后，硬盘上的哪个文件会消失。
2.  **预判用户的困惑**：浏览器的 File System Access API 出于安全考虑，必须要求用户手动授权。为了不让用户面对突如其来的权限弹窗感到惊慌，我们在模态框里加了一行温馨提示，解释了"为什么浏览器会弹窗"以及"你应该点哪里"。

这种对细节的执着，虽然增加了开发成本，但它让工具变得不再冰冷，而是充满了对使用者的尊重。

## 二、 谋定而后动：逻辑理清了，代码才写得对

在开发初期，我曾试图在**文章详情页**放置删除按钮。逻辑似乎很顺：看完文章，觉得不好，顺手删了。

但在实际推演中，我发现了一个逻辑黑洞：
> 如果用户在详情页删除了这篇文章，那么删除成功后的下一秒，他面前展示的这个页面（详情页）所依赖的数据源就消失了。

这就像是**"抽走了脚下的地毯"**。

于是，我们及时叫停了方案，重新审视了交互流程。最终，我们决定将删除入口移至**首页列表**。
*   **理由**：在列表中删除一项，只是列表少了一行，用户依然停留在列表页，上下文环境是稳定的。

这次调整让我明白，写代码前先理清业务逻辑和状态流转，往往能避免方向性的错误。**慢即是快**。

## 三、 大局观：不造轮子，善用生态

在设计"发布文章"功能时，我们面临一个诱惑：要不要在博客里做一个全功能的 Markdown 编辑器？

如果这样做，工程量巨大，而且很难超越 Typora 这样成熟的本地编辑器。

**"我们的目标是什么？"**
我们的目标是**展示**和**管理**，而不是**编辑**。

基于这个大局观，我们确立了**"Typora + Blog"**的最佳实践方案：
1.  **专业的事交给专业的人**：写作和编辑依然在 Typora 中完成，利用其强大的 Frontmatter 支持。
2.  **博客只做连接者**：博客系统只负责"读取"、"预览"和"发布"。
3.  **标准化**：我们利用 Frontmatter 中的 `title` 字段自动生成规范的文件名，既利用了现有的格式标准，又解决了文件命名混乱的问题。

这种"不盲目做加法"的克制，反而让整个系统更加轻量、灵活，也更符合开发者的使用习惯。

## 结语

今天的开发不仅仅是功能的堆砌，更是一次关于**"如何把产品做好用"**的实践。

从删除指引的细微优化，到交互流程的逻辑重构，再到技术选型的取舍，每一步都体现了**"以人为本"**的开发理念。

代码是逻辑的载体，而好的产品，是逻辑与人性的共鸣。
